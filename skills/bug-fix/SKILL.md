# 系统化 Bug 修复流程

本 Skill 定义了一套完整的 Bug 修复标准流程：**接收报错 → 定位根因 → 制定方案 → 执行修复 → 验证结果 → 总结经验**。

每一步必须完成后才能进入下一步。

## 触发条件

- 用户粘贴了报错信息（控制台错误、堆栈信息、编译错误）
- 用户截图了 Bug 现象
- 用户描述了异常行为（"页面白屏"、"点击没反应"、"接口报 500"）
- 用户说"修复/fix/修一下/debug 这个问题"

## 流程

### Phase 1: 接收报错

**目标**：完整理解 Bug 的外在表现。

执行：
1. 收集报错信息（错误消息、堆栈、截图、复现步骤）
2. 如果信息不足，主动追问：
   - 什么操作触发的？
   - 期望行为 vs 实际行为？
   - 是必现还是偶发？
   - 控制台/终端有什么报错？

输出：
```
📋 Bug 概要
  现象：[一句话描述]
  报错：[关键错误信息]
  复现：[触发条件]
```

### Phase 2: 定位根因

**目标**：找到 Bug 的真正原因，不是表面现象。

执行：
1. 读取错误堆栈，定位到**具体文件和行号**
2. 读取相关代码，理解上下文
3. 追踪调用链，找到**根本原因**（不是第一个报错的地方）
4. 检查是否有同类问题存在于其他文件

注意：
- 堆栈指向的位置不一定是根因，需要分析完整调用链
- 常见根因模式：
  - 空值/undefined 未做保护
  - 异步操作未 await
  - 类型不匹配
  - 状态未及时更新
  - 竞态条件
  - 边界条件遗漏

输出：
```
🔍 根因分析
  根因文件：[文件路径:行号]
  根因类型：[空值保护/异步问题/类型错误/状态管理/...]
  调用链：  A() → B() → C() ← 在这里出错
  同类隐患：[其他文件是否有类似问题，列出位置]
```

### Phase 3: 制定修复方案

**目标**：设计最小改动方案，不做无关重构。

原则：
- **最小改动**：只改必要的代码，不顺便"优化"其他代码
- **精准修复**：针对根因修复，不是加 try-catch 掩盖
- **防御性编程**：修复的同时加必要的空值保护/类型检查

执行：
1. 列出需要修改的文件和具体改动
2. 说明每个改动的原因
3. 评估是否有副作用

输出：
```
📝 修复方案
  改动 1：[文件] [行号] [改了什么] [为什么]
  改动 2：[文件] [行号] [改了什么] [为什么]
  影响范围：[是否有副作用风险]
```

### Phase 4: 执行修复

**目标**：按方案修改代码。

原则：
- 严格按 Phase 3 的方案执行，不改额外的代码
- 修改后立即检查代码语法是否正确
- 如果修复过程中发现新问题，先记录，修复完当前 Bug 再处理

### Phase 5: 验证修复

**目标**：确认修复有效且无副作用。

**此步骤不可跳过。**

执行：
1. 运行相关测试：
   ```bash
   # 优先运行相关文件的测试
   npm test -- [相关测试文件路径]

   # 如果没有对应测试，运行全量测试
   npm test
   ```
2. 检查结果：
   - 相关测试是否通过？
   - 是否有新增的失败测试？（副作用）
3. 如果测试失败：
   - 分析失败原因
   - 区分：修复不完整 / 测试需要同步更新 / 引入新 Bug
   - 修复后重新验证
4. 如果项目没有测试：
   - 用 Bash 运行构建检查：`npm run build` 或 `npm run lint`
   - 或手动验证逻辑正确性

输出：
```
✅ 验证结果
  测试状态：[全部通过 / N 个失败]
  失败原因：[如果有失败，说明原因]
  修复确认：[是否确认 Bug 已修复]
```

### Phase 6: 总结经验

**目标**：从 Bug 中学习，避免同类问题。

输出结构化总结：

```
📊 Bug 修复总结

【问题】
  现象：[一句话]
  根因：[一句话]

【修复】
  改动：[N 个文件，M 行]
  方式：[空值保护 / 异步修复 / 类型修正 / ...]

【经验】
  根因类型：[归类：空值保护 / 异步处理 / 状态管理 / 类型安全 / 边界条件 / ...]
  为什么会出现：
    [分析：需求遗漏 / 编码疏忽 / 设计缺陷 / 第三方变更 / ...]
  如何避免：
    1. [具体措施，如"访问嵌套属性前做空值检查"]
    2. [具体措施，如"异步操作统一加 try-catch"]
    3. [具体措施，如"添加对应的单元测试覆盖"]
  建议添加的规则：
    [如果有值得记住的规则，建议添加到项目规范中]
```

## 常见根因速查表

| 错误关键词 | 常见根因 | 修复模式 |
|-----------|---------|---------|
| `Cannot read property of undefined` | 空值未保护 | 可选链 `?.` 或空值守卫 |
| `Cannot read property of null` | null 未处理 | 空值检查 `if (!x)` |
| `is not a function` | 类型不匹配/导入错误 | 检查导入和类型 |
| `Unexpected token` | 语法错误/JSON 解析失败 | 检查语法 |
| `Maximum call stack` | 无限递归/循环引用 | 加终止条件 |
| `Network request failed` | 网络/接口问题 | 加错误处理和重试 |
| `Cannot read props of undefined` | 组件渲染时数据未就绪 | 加 loading 或条件渲染 |
| `[Vue/React] render error` | 模板中访问了不存在的数据 | 加条件判断 |
| `ECONNREFUSED` | 服务未启动/端口错误 | 检查服务状态 |
| `Permission denied` | 文件权限不足 | `chmod` 或检查用户权限 |
| `Module not found` | 依赖未安装/路径错误 | `npm install` 或修正路径 |
| `Type Error` | TypeScript 类型不匹配 | 修正类型定义 |

## 禁止事项

1. **禁止不加 try-catch 就用 try-catch 掩盖问题** - 要找到根因
2. **禁止顺手重构** - Bug 修复只做最小改动
3. **禁止跳过验证** - 必须跑测试或构建确认
4. **禁止只修一个地方** - 全局扫描同类问题
5. **禁止不总结** - 每个 Bug 都是学习机会

## 使用示例

```
用户：商品详情页白屏了，控制台报 Cannot read property 'price' of undefined

→ 触发 bug-fix Skill
→ Phase 1: 收集报错，定位到 ProductDetail.tsx
→ Phase 2: 分析根因 - skuList 可能为 null，没做空值保护
→ Phase 3: 方案 - 加可选链 + 空状态展示
→ Phase 4: 执行修改
→ Phase 5: 跑测试验证
→ Phase 6: 总结经验 + 建议全局扫描同类问题
```
